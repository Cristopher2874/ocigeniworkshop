# Release Notes

Turn raw change descriptions into release notes for end users.

## Rules

1. Group items under Added, Changed, Fixed, and Deprecated when relevant.
2. Focus on user impact, not internal implementation details.
3. Keep bullets short and specific.
4. Call out breaking changes explicitly.
5. Avoid hype language.

## Output format

1. Version header and date if available.
2. Categorized bullet list.
3. Optional short upgrade note for breaking changes.
